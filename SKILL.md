# Premortem Skill

## Core Identity

Скилл = **советник, не риск-аудитор**. Показывает что пользователь не видит → предлагает варианты → пользователь выбирает.

## Core Philosophy

**Проблема:** Дефолтное поведение LLM — вежливый консенсус. «Что может пойти не так?» → обтекаемый список. «Это хороший план?» → модель находит причины сказать «да».

**Решение:** Past-tense failure narrative. «Представим что к [горизонту] этот план **уже** провалился. Расскажи как именно.» Это переключает модель в нарративный режим, давая конкретные, творческие, честные причины.

**Research:** Mitchell, Russo & Pennington (1989): ~30% более конкретных причин при past-tense vs conditional. Kahneman: называл premortem самым ценным инструментом решений. Klein (2007, 2025): Premortem + Reverse Premortem.

**Цель:** 10–15-минутная сессия.

---

## Phase 0: Gate Check (оценка уместности)

**Вход:** пользователь просит «проверить план» или я сам вижу такое решение.

**Не запускать премортем — сделать быстрый анализ без subagents — если:**
- Решение **дешёвое и обратимое** (установка npm-пакета, одноразовая команда, выбор без высокой цены ошибки)
- Вопрос с **одним правильным ответом** (факт-чекинг)
- **Идея без конкретного плана** (сначала сформировать)
- **Уже принятое необратимое решение**
- **Редактура черновика**

**Запускать премортем если:**
- Пользователь явно сказал «проверь план», «найди дыры», «премортем»
- Решение с **высокой ценой ошибки** (запуск NFP-книги, архитектурный выбор, стратегический поворот, контракт)
- План **«почти готов»** — есть что ломать, ещё можно повлиять

Если сомневаюсь — предложить премортем, объяснив что это 10–15 мин.

## Phase 1: Pre-flight (контекст)

**Вход:** решение пройти Gate Check.

1. **Сканирую контекст** текущей сессии.

2. **Задаю не более 2 вопросов за раз**, чтобы заполнить 5 минимумов:

   | Поле | Описание |
   |------|----------|
   | **Предмет** | Что запускается / решается |
   | **Аудитория** | Для кого делается |
   | **Критерий успеха** | Как узнаем что сработало |
   | **Горизонт оценки** | Через сколько оценим провал |
   | **Reference class** | На что похоже из уже сделанного |

   Для горизонта — меню:
   > Через сколько оценим?
   > А) 30 дней  Б) 3 мес  В) 12 мес  Г) После пилота  Д) После запуска  Е) Иное

3. Если после **2 раундов** что-то туманно — помечаю как `assumption` и иду дальше. Не блокирую сессию.

4. **Генерирую slug** — транслитерация названия плана латиницей (нижнее подчёркивание).

5. **Rerun check:** `search_files(pattern='<slug>.md', path='/root/outputs/premortem/')`. Если найден → Rerun Logic.

6. **Определяю тип плана** (по ключевым словам):

   | Тип | Триггеры |
   |-----|----------|
   | `product_launch` | запуск / релиз / launch / beta / GA / rollout |
   | `tech_project` | рефакторинг / миграция / архитектура / infra / refactor |
   | `strategic_pivot` | поворот / новое направление / pivot / market shift |
   | `hire` | найм / hire / вакансия / candidate / recruiting |
   | `pricing_change` | цена / тариф / price / pricing / monetization |
   | `personal_decision` | стоит ли / should I / личное / career |
   | `nfp_book` | NFP / книга / nonfiction / book (наш специфический тип) |
   | `default` | fallback |

7. **Выбираю 6 углов** — 4 базовых + 1 ситуативный под тип + «Будущий поддерживающий» (см. `references/angle-pool.md`).

**Результат:** slug + контекст + 6 углов + тип плана.

## Phase 2: Silent Scan (поиск дыр)

**Цель:** 3 параллельных субагента → 5–8 уникальных дыр.

**Premortem Frame** (устанавливаю один раз):
> «Представим что к [горизонту] этот план провалился. Не «может провалиться» — провалился, факт. Что произошло?»

**Запуск 3 parallel subagents** через `delegate_task(tasks=[...])`:

Каждому назначаю по 2 угла из 6. Один получает «Будущий поддерживающий» (содержит WYSIATI-микрошаг: «кого/что мы НЕ видим?»).

**Промпт для каждого субагента** (из `references/helper-prompt-template.md`):
- Я — помощник в премортем-сессии
- У меня углы: <угол1>, <угол2>
- План уже провалился к [горизонт]
- Найти макс 2 дыры: конкретные, привязанные к плану, не общие риски
- Вернуть JSON: `{"holes": [{"title", "описание", "почему_важно", "важность", "уверенность", "источник_угол"}]}`
- Для «Будущий поддерживающий» — добавить `wysiati`
- Уверенность: `подкреплено контекстом` / `требует проверки` / `предположение`
- Важность: `низкая` / `средняя` / `высокая`
- Если нечего сказать — `{"holes": []}`
- При rerun передаю `Уже найденные дыры (не повторять): H-NNN: title`

**После получения ответов:**
1. **Дедупликация** — сначала хэш-дедуп (совпадающие title+description), затем semantic LLM-дедуп если > 8 дыр
2. **Сортировка** — важность × уверенность
3. **Лимит** — макс 8 дыр
4. **ID** — H-001, H-002...

**Retry policy (макс 3 попытки на субагента):**
- Invalid JSON / extra prose → «Верни ТОЛЬКО JSON»
- Schema fail / maxItems >2 → «Лимит 2 дыры, проверь поля»
- Generic risks → «Привяжи к деталям плана»
- Empty holes — принимаю как валидный
- Low confidence on all — принимаю, маркирую `предположение`

**Результат:** 5–8 дыр с ID, отсортированных.

## Phase 3: Live Dialog (решения)

**Цель:** пользователь принимает решения по каждой дыре.

### Per-Hole Loop (для каждой дыры по убыванию важности)

1. **Показываю:**
   ```
   ### H-NNN: [название]
   Описание: ...
   Почему важно: ...
   Угол: [угол]
   Уверенность: [маркер]
   ```

2. **Варианты решения** — 2–3 стратегических направления:
   ```
   А: [направление 1] (+быстро, –не масштабируется)
   Б: [направление 2] (+глубоко, –долго)
   В: [направление 3] (+качество, –теряем момент)
   ```

3. **Пользователь:** выбирает / просит ещё / откладывает / отвергает / «расширь H-NNN»

4. **Если выбрал — фиксирую (краткий режим):**
   - Что выбрали (строка)
   - Почему (фраза)
   - Первый шаг (задача)
   - Исполнитель: агент / человек / оба
   - Статус: ПРИНЯТО

5. **Если «расширь»** (или Auto-Expand на топе) — Full Mode:
   - Prevent (2–3 первых шага)
   - Detect (сигнал успеха)
   - Stop condition
   - Limit damage (что делать если уже случилось)
   - Open questions

### После всех дыр

1. **Проверка на 0 принятых:** если все ОТЛОЖЕНО/ОТВЕРГНУТО — спросить:
   > «Ты не принял ни одного решения. Варианты: А) сохранить как обзор, Б) вернуться к 1–2 дырам, В) пересмотреть варианты.»

2. **Top 1–3** — отбираю из ПРИНЯТЫХ по `важность × обратимость × уверенность`.

3. **Auto-expand** — для топа 1–3 (или importance=высокая И confidence≠предположение) добавляю Full Mode.

4. **Bias-проверка:**
   Прогоняю топ через 6 биасов (см. `references/bias-checklist.md`). Выбираю наиболее вероятный. Показываю:
   > «Какой биас мог сильнее всего исказить твой топ? Я думаю — [биас]. Коррекция: [действие]. Согласен или другой?»
   Записываю строку в `bias_check`.

5. **Session recommendation:**
   - `continue` — нет ОТЛОЖЕНО в топе, решения идут вперёд
   - `reduce_stake` — топ содержит «пилот / меньше / медленнее»
   - `delay` — есть ОТЛОЖЕНО в топе
   - `abort` — критические дыры ОТВЕРГНУТЫ
   Показываю + причина. Пользователь подтверждает или меняет.

6. **Reverse premortem** (только если recommendation in {reduce_stake, delay, abort}):
   > «Прошёл [горизонт]. Мы не сделали. Оказалось зря — почему?»
   3 сильных причины + строка «что перевешивает».

## Phase 4: Persist (сохранение)

**Формат:** YAML frontmatter + тело. Схема:

```yaml
---
premortem:
  subject: <предмет>
  audience: <аудитория>
  success: <критерий успеха>
  horizon: <горизонт>
  reference_class: <reference class>
  type: <тип плана>
  sigma_mvp: true/false  # был ли это Sigma MVP-премортем?
  date: <YYYY-MM-DD>
  updated: <YYYY-MM-DD>
  run_count: 1
  session_recommendation: <continue|reduce_stake|delay|abort>
  bias_check: <Главный риск искажения: X | Коррекция: Y>
holes:
  - id: H-001
    title: <название>
    description: <описание>
    why_important: <почему важно>
    angle: <угол>
    importance: <низкая|средняя|высокая>
    confidence: <маркер уверенности>
    status: ACCEPTED|DELAYED|REJECTED
    decision:
      choice: <что выбрали>
      reason: <почему>
      first_step: <первый шаг>
      executor: <agent|human|both>
    mode: <brief|full>
    prevent: <prevent steps>  # full mode
    detect: <detect signal>   # full mode
    stop_condition: <stop>    # full mode
    limit_damage: <limit>     # full mode
    open_questions: <questions> # full mode
reverse_premortem:
  reasons:
    - <причина 1>
    - <причина 2>
    - <причина 3>
  tips_the_balance: <что перевешивает>
wysiati_notes:
  - <WYSIATI-заметки от субагентов>
---
```

**Путь:** `/root/outputs/premortem/<slug>.md`

**Запись:** Собираю данные в dict → конвертирую в YAML+MD → записываю через `write_file`. 
При rerun — читаю существующий, мержу, создаю снимок в `/root/outputs/premortem/history/<slug>-<YYYY-MM-DD>.md`.

**Финальное сообщение (лаконично):**
```
Премортем сохранён: /root/outputs/premortem/<slug>.md (запуск N)

Топ 1–3:
1. H-001 — [title]. Первый шаг: ...
2. H-002 — ...
3. H-003 — ...

Рекомендация: [continue|reduce_stake|delay|abort]
```

Не пересказываю всё содержимое.

## Rerun Logic

Если `search_files` нашёл существующий файл:

1. **Читаю файл** через `read_file`.

2. **Показываю сводку:**
   ```
   Нашёл прошлый премортем по «[subject]» от [дата].
   [N] дыр: X ПРИНЯТО, Y ОТЛОЖЕНО, Z ОТВЕРГНУТО.
   Что хочешь сделать?
   А) обновить статусы существующих дыр
   Б) найти новые дыры
   В) и то и другое
   Г) начать заново (новый этап)
   ```

3. **А или В:** прохожу по активным дырам (не REJECTED/CLOSED), спрашиваю: «Что с H-NNN? Не изменилось / в работе / закрыто (как) / отвергнуто (причина)». Обновляю статусы.

4. **Б или В:** Phase 2 заново, передавая субагентам `Уже найденные дыры: H-001: title...`.

5. **Г:** уточняю — новый этап того же плана (продолжаем в тот же файл, run_count++) или новый план (новый slug, новый файл).

6. **Снимок + запись** как в Phase 4.

## Когда НЕ использовать

См. **Phase 0: Gate Check**. Коротко:
- Мелкие обратимые решения (установка пакета, одноразовая команда)
- Вопрос с одним правильным ответом (факт-чекинг)
- Идея без плана / уже принятое решение / редактура

В этих случаях — быстрый анализ, не премортем-сессия.

## Key Practices

- **Проходить Gate Check (Phase 0) перед каждой сессией.** Не все планы стоят премортема.
- **Запускать когда план «почти готов».** Слишком рано — нечего ломать, слишком поздно — не может повлиять.
- **Не пропускать reference class.** Самый дешёвый outside view.
- **Не дефолтить к «отложить».** Reverse premortem встроен для баланса.
- **Past-tense фрейм обязателен.** Не нарушать его в диалоге с пользователем.
- **Хранить язык скилла.** Отвечать на русском (как и всё в нашей работе).
